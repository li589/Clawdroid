**可优化点**
- 状态粒度过粗：`OverviewUiState` 任何字段变化都会推动整个 Overview 分支重组，尤其 1 秒一次的 `dashboardMetrics` 会连带大量卡片刷新；建议把仪表盘、截图、事件日志拆成更细的 `StateFlow` 或独立子状态。
- 采样开销偏高：当前每秒都读多份 `/proc`，Root 可用时还额外起一次 `su -c`，见 [DashboardMetrics.kt:L63-L120](file:///d:/temp_desktop/Proj/Clawdroid/ClawApp/app/src/main/java/com/clawdroid/app/ui/DashboardMetrics.kt#L63-L120)；建议将 Root 进程采样降频到 2~5 秒，或与事件流结合做按需刷新。
- 轮询与事件流重复：仪表盘里 `windowSummary/daemonMetrics/runtimeMetrics` 来自事件流，而 CPU/内存来自轮询，当前是双通道并行；若运行时已能推送更多诊断字段，可把部分轮询指标下沉到事件流，减少 IO。
- 图片状态过重：`latestCapturePreview` 直接放在主 `OverviewRuntimeState` 中，解码后会带着大对象参与状态复制与 UI 更新；建议改为独立 `MutableStateFlow<ImageBitmap?>` 或只存 `ByteArray/path`，在预览卡内部异步解码。
- 截图链路是“两步操作”：先 `captureScreen()` 再 `readLatestCapture()`；UI 上会出现两次状态更新，建议增加“截图并预览”合并动作，减少中间态和重复重组。
- `eventLines` 每次 prepend 新列表，见 [OverviewController.kt:L523-L528](file:///d:/temp_desktop/Proj/Clawdroid/ClawApp/app/src/main/java/com/clawdroid/app/ui/OverviewController.kt#L523-L528)；高频事件下会持续分配新 List，建议改 ring buffer 或 `SnapshotStateList`。
- `MetricInfoCard` 固定宽度 `156.dp`，见 [ConsoleComponents.kt:L1047-L1079](file:///d:/temp_desktop/Proj/Clawdroid/ClawApp/app/src/main/java/com/clawdroid/app/ui/ConsoleComponents.kt#L1047-L1079)；在窄屏/折叠屏上可改为自适应权重或 `Grid`，减少换行抖动。
