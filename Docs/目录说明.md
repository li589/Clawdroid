# Clawdroid 目录说明

## 目录目标

本说明用于解释当前仓库的顶层目录划分，明确哪些目录属于 Clawdroid 主工程，哪些目录属于正式文档，哪些目录只是第三方参考资料，从而避免继续把 Brain Host、Runtime Executor 与外部调研源码混在一起理解。

## 顶层目录

### `ClawApp/`

用于存放 Android 客户端相关内容，代表面向用户安装的 APK 工程。

当前应将其理解为 `Clawdroid App` 的源码目录，也就是整个系统的 Brain Host。

当前包含：

- `settings.gradle.kts`
  - `ClawApp` 独立 Gradle 工程入口
- `build.gradle.kts`
  - `ClawApp` Gradle 根插件版本声明
- `gradle.properties`
  - `ClawApp` Android/Gradle 构建属性
- `local.properties`
  - Android SDK 本地路径
  - 可选包含 `clawdroid.runtime.sharedSecret`
  - 属于本地开发文件，不应作为主仓共享配置
- `app/`
  - Android App 主模块
  - 包含 UI、`AccessibilityService`、编排器、工具执行层、运行时客户端、LSPosed/Xposed 入口
- `xposed-stubs/`
  - 本地编译桩模块
  - 用于在不直接依赖外部 Xposed API 仓库的情况下完成编译期接口引用

### `ClawRuntime/`

用于存放底层执行器与 Magisk 模块相关内容，代表高权限运行时工程。

当前物理目录名已统一为 `ClawRuntime/`。

当前包含：

- `runtime/`
  - Go 守护进程工程
  - 当前承载 UDS 服务端、协议处理、鉴权、审计和受限执行能力
- `magisk/`
  - Magisk 模块打包目录
  - 包含 `module.prop`、`service.sh`、`sepolicy.rule`、配置文件与 `webroot/` 状态快照产物
- `scripts/`
  - 打包、编译、分发辅助脚本目录
  - 包含 `build-runtime.ps1`、`build-magisk.ps1`、`sync-shared-secret.ps1`

### `Docs/`

用于存放项目正式文档，不与代码目录混放。

当前包含：

- `基础方案设计.md`
- `protocol.md`
- `threat-model.md`
- `目录说明.md`
- `真机安装验收清单.md`
- `发布前清单.md`
- `compatibility-matrix.md`
- `release-notes.md`

其中 `基础方案设计.md.bak` 仅作为历史备份文件保留，不作为当前正式文档入口。

### `GitSource/`

用于存放外部参考源码与调研资料。

该目录的定位必须明确为：

- 不是当前主工程实现目录
- 不是主仓构建前提
- 不是发行交付物
- 不受主仓根级 `MIT` 许可证重新授权

当前保留该目录的目的主要是：

- 对比 Root / Magisk / Zygisk / LSPosed 相关上游实现
- 为后续兼容性调研和风险评估提供参考

第三方许可证边界说明见仓库根 [THIRD_PARTY_NOTICES.md](file:///d:/temp_desktop/Proj/Clawdroid/THIRD_PARTY_NOTICES.md)。

## 当前推荐理解方式

从项目视图上看，仓库应理解为：

1. `ClawApp` 目录承载的是 `Clawdroid App`，它是用户层、AI 层和工具编排层的总入口
2. `ClawRuntime` 目录是 Root 执行侧和 Magisk 打包侧宿主
3. `Docs` 是架构、协议、安全和目录规范的正式文档区
4. `GitSource` 是受边界约束的参考资料区，不参与主仓功能归属判断

补充说明：

1. 共享密钥推荐由仓库根 `local.properties` 统一保存，通过构建脚本同步到 App 与 Magisk 模块；该文件属于本地私有配置。
2. `ClawRuntime/magisk/webroot` 当前承载的是本地诊断快照，不是长期开放的远程控制台。

## App 内部分层建议

在 `Clawdroid App` 内部，当前推荐继续沿着三层组织演进：

1. `orchestrator`
   - 负责把聊天文本、规则或未来的模型工具调用，解析成明确的工具意图
2. `tools`
   - 负责把页面确认、截图、Shell、输入注入等能力显式建模为工具，并统一执行结果
3. `runtime`
   - 负责对接 `ClawRuntime` IPC 客户端，屏蔽底层旧命名和协议细节

这样划分后，项目表面结构会更清晰，也更符合“两个主体程序 + 一组正式文档 + 一组第三方参考资料”的组织目标。
