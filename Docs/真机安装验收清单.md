# ClawRuntime 真机安装验收清单

## 1. 验收目标

用于验证 `ClawRuntime` Magisk 模块在真机上的安装、启动和基础自检是否符合当前规范。

当前规范：

- 模块 ID：`clawruntime`
- 运行时二进制：`clawdroid-runtime`
- 配置文件：`config/runtime.yaml`
- 仓库内模板文件：`ClawRuntime/magisk/config/runtime.yaml`
- 本地生成文件：`ClawRuntime/magisk/config/runtime.generated.yaml`
- 不再提供旧模块 ID 或旧配置文件名的自动迁移兼容

## 2. 验收前准备

- 设备已 Root，且安装 `Magisk` 或兼容的模块管理器
- 设备已开启 `adb`
- 准备待安装包：`ClawRuntime/dist/ClawRuntime-magisk.zip`
- 确认仓库根 `local.properties` 或环境变量中已经提供 `clawdroid.runtime.sharedSecret`
- 如需初始化本地配置，可从 [local.properties.example](file:///d:/temp_desktop/Proj/Clawdroid/local.properties.example) 复制
- 若本轮是发布包验收，需提前准备 `auth.allowed_signatures` 的正式签名摘要白名单
- 如设备上仍保留旧版本模块，先手动备份配置并卸载旧模块，再安装当前包

建议记录以下基础信息：

- Android 版本
- ROM/厂商版本
- Root 方案与版本
- 模块管理器版本
- 当前是否存在旧模块残留目录

- 发布包场景下 `auth.allowed_signatures` 非空

- 结果回填到 [compatibility-matrix.md](file:///d:/temp_desktop/Proj/Clawdroid/Docs/compatibility-matrix.md)

## 3. 场景 A：全新安装

### 安装步骤

1. 在模块管理器中安装 `ClawRuntime-magisk.zip`
2. 安装完成后不要立刻重启，先查看安装日志
3. 重启设备
4. 重启完成后执行安装后自检：

```bash
adb shell su -c 'sh /data/adb/modules/clawruntime/verify.sh'
```

### 预期结果

- 安装日志中显示 `Canonical module ID: clawruntime`
- `/data/adb/modules/clawruntime/module.prop` 中 `id=clawruntime`
- `/data/adb/modules/clawruntime/service.sh`、`post-fs-data.sh`、`customize.sh`、`verify.sh` 均存在
- `/data/adb/modules/clawruntime/bin/clawdroid-runtime` 存在且可执行
- `/data/adb/modules/clawruntime/config/runtime.yaml` 存在
- 安装包中的 `config/runtime.yaml` 应来自本地生成配置，而不是仓库模板占位内容
- `verify.sh` 输出 `verify passed: 0 failure(s), N warning(s)` 或更严格的同格式通过信息
- `service.sh` 拉起运行时后，日志文件存在：

```bash
adb shell su -c 'ls -l /data/local/tmp/clawdroid/logs/runtime-service.log'
```

- 模块 `webroot/` 下已生成排查快照：

```bash
adb shell su -c 'ls -l /data/adb/modules/clawruntime/webroot'
adb shell su -c 'cat /data/adb/modules/clawruntime/webroot/status.json'
adb shell su -c 'cat /data/adb/modules/clawruntime/webroot/verify.json'
```

## 4. 场景 B：从旧包手动切换

### 前置条件

- 设备中可能存在旧版本模块目录
- 旧模块中的配置若仍需保留，需要先手动备份

### 安装步骤

1. 先确认设备上的旧模块目录，再手动备份旧配置，例如：

```bash
adb shell su -c 'ls -l /data/adb/modules'
```

2. 在模块管理器中卸载旧版本模块
3. 确认旧模块目录已清理，或至少不会继续参与开机加载
4. 安装 `ClawRuntime-magisk.zip`
5. 如有需要，把旧配置内容手动转换并写入 `/data/adb/modules/clawruntime/config/runtime.yaml`
6. 重启设备后执行：

```bash
adb shell su -c 'sh /data/adb/modules/clawruntime/verify.sh'
```

### 预期结果

- 新模块目录存在：`/data/adb/modules/clawruntime`
- 新模块不依赖任何 `migration-state` 目录
- 若旧模块目录仍残留，不应影响 `clawruntime` 自检通过
- 当前配置文件仅使用 `runtime.yaml`

## 5. 关键文件检查

可手动核对以下文件：

```bash
adb shell su -c 'cat /data/adb/modules/clawruntime/module.prop'
adb shell su -c 'ls -R /data/adb/modules/clawruntime'
adb shell su -c 'cat /data/adb/modules/clawruntime/config/runtime.yaml'
```

重点观察：

- 模块 ID 是否为 `clawruntime`
- `clawdroid-runtime` 是否存在
- `runtime.yaml` 是否存在且内容符合预期
- `runtime.yaml` 中是否包含：
  - `auth.shared_secret`
  - `auth.allowed_packages`
  - `auth.allowed_signatures`
  - `capability.screenshot_enabled`
  - `capability.file_bridge_enabled`
- 仓库模板 `ClawRuntime/magisk/config/runtime.yaml` 不应保留真实共享密钥
- 目录中不应再依赖 `migration-state`
- `webroot/status.json`、`webroot/verify.json`、日志 tail 文件是否存在

## 6. 回归检查

安装完成后，至少再验证一次以下结果：

- 设备重启后模块仍处于启用状态
- `runtime-service.log` 中出现启动记录
- `clawdroid-runtime` 二进制可执行
- App 侧可继续连接本地 IPC
- App 概览页可显示本地环境、权限修复入口与实时仪表盘
- App 内可完成 `Runtime Probe`、`Capabilities`、截图预览与至少一次输入动作
- `status.json` 中 `runtime_state`、`verify_exit_code`、`updated_at` 字段有更新
- `/data/local/tmp/clawdroid/logs` 与 `/data/local/tmp/clawdroid/audit` 目录存在

## 6.1 LSPosed 专项验收

当设备已安装 LSPosed，且需要验证 `lsposed_runtime_loaded=true` 的真实注入场景时，除基础回归外，还应额外完成本节。

### 目标

- 明确区分“LSPosed 已安装”和“模块已真实注入”
- 将模块启用、作用域配置、Marker 校验和能力复验沉淀为固定步骤
- 避免仅依赖当前设备的历史配置或偶发残留状态

### 前置检查

先确认以下前提满足：

- `ClawApp` 安装包包含 `xposed_init` 资源，且入口类为 `com.clawdroid.app.xposed.ClawdroidXposedInit`
- 当前安装包名与模块作用域目标一致
- `ClawRuntime` 已按前述步骤安装并通过 `verify.sh`
- App 当前签名摘要已同步到设备端 `auth.allowed_signatures`

可选检查命令：

```bash
adb shell pm path com.clawdroid.app.debug
adb shell su -c 'ls -ld /data/adb/lspd /data/adb/modules/zygisk_lsposed /data/adb/modules/riru_lsposed /data/adb/modules/lsposed'
adb shell su -c 'cat /data/adb/modules/clawruntime/config/runtime.yaml | grep allowed_signatures -n'
```

### 推荐启用流程

1. 在 LSPosed 中确认 `ClawApp` 已被识别为模块
2. 显式启用模块，不要假设“安装后默认启用”
3. 为当前包名添加作用域；调试包使用 `com.clawdroid.app.debug -> com.clawdroid.app.debug` 自作用域
4. 完成模块启用和作用域设置后，强制停止目标 App，再重新启动
5. 回到 App 概览页，刷新本地环境并执行 `Capabilities`
6. 如已开启事件订阅，再复验一次 `capability_changed`

### 当前设备已验证流程

以下流程已在当前首台回归设备 `Xiaomi Pad 6 / Android 13 / MIUI V14.0.10.0.TMZCNXM / Magisk 27001` 上复测通过，可作为项目内标准流程保存：

1. 先确认基础链路已通过：
   - `ClawRuntime` 最新模块已安装并重启
   - `verify.sh` 返回 `0 failure(s), 0 warning(s)`
   - `Runtime Probe` 已经是 `Ready`
2. 确认设备上存在 LSPosed 框架目录：

```bash
adb shell su -c 'ls -ld /data/adb/lspd /data/adb/modules/zygisk_lsposed'
```

3. 若设备上看不到标准 `LSPosed Manager` 包，不要直接判定“LSPosed 不可用”；先确认是否存在可替代的寄生管理入口或用户自备的 LSPosed 配置入口
4. 在可用的 LSPosed 管理入口中完成两件事：
   - 显式启用 `ClawApp` 模块
   - 为 `com.clawdroid.app.debug` 添加自作用域，使模块代码真正注入 `com.clawdroid.app.debug` 自己的进程
5. 当前设备实测中，仅完成授权后立即冷启动 App 仍可能看不到 marker；完成模块启用与自作用域后，必须重启设备
6. 设备重启完成后，强制停止并冷启动 `com.clawdroid.app.debug`

```bash
adb shell am force-stop com.clawdroid.app.debug
adb shell cmd activity start-activity -W -n com.clawdroid.app.debug/com.clawdroid.app.MainActivity
```

7. 冷启动后立刻验证 marker 是否落地：

```bash
adb shell su -c 'cat /data/user/0/com.clawdroid.app.debug/files/xposed_runtime_marker.json'
```

8. 预期 marker 至少包含以下字段：
   - `xposed_injected=true`
   - `process_name="com.clawdroid.app.debug"`
   - `package_name="com.clawdroid.app.debug"`
9. 再执行 `probe` 或 `capabilities` 复验，预期返回：
   - `lsposed=true`
   - `lsposed_runtime_loaded=true`
   - `lsposed_runtime_process="com.clawdroid.app.debug"`
   - 会话状态仍为 `Ready`

### 当前设备注意事项

- 当前设备已检测到 LSPosed 框架目录，但未稳定检测到标准 `LSPosed Manager` 包；因此“管理入口存在”与“标准 Manager 包存在”不是同一个判断条件
- 在这台设备上，若管理入口中根本看不到 `com.clawdroid.app.debug` 自身，通常意味着当前使用的不是正确的 LSPosed 管理入口，或模块尚未被正确识别/启用
- 单纯给其他 App 随机添加作用域，对 `ClawApp` 自身 marker 落地没有帮助；当前实现依赖 `com.clawdroid.app.debug` 自进程注入，不能用“别的目标 App 已加作用域”替代
- 当前设备实测中，LSPosed 相关授权调整后需要重启设备，不能只依赖热重启目标 App
- 若设备使用 `Kitsune Mask` 或其他手动授权 Root 管理器，不要把首次 `Permission denied` 直接解读为“模块不兼容”；应先确认当前 App 是否已在超级用户列表中被允许，然后再继续做 Runtime 会话探针

### Runtime 业务闭环复测

以下步骤用于在“重装最新 App + 最新模块”后快速判断当前设备到底是 `Ready`、`Degraded`、还是鉴权失败：

1. 安装最新 `app-debug.apk`
2. 安装最新 `ClawRuntime-magisk.zip` 并重启设备
3. 先确认模块运行态：

```bash
adb shell su -c 'cat /data/adb/modules/clawruntime/webroot/status.json'
adb shell su -c 'cat /data/adb/modules/clawruntime/webroot/verify.json'
```

4. 推荐优先使用统一桥接脚本执行 `probe / capabilities / chat_prompt`，避免每次手写长命令：

```bash
pwsh -File .\scripts\invoke-debug-runtime-operation.ps1 -DeviceId 7710a01f -Operation probe
pwsh -File .\scripts\invoke-debug-runtime-operation.ps1 -DeviceId 7710a01f -Operation capabilities
pwsh -File .\scripts\invoke-debug-runtime-operation.ps1 -DeviceId 7710a01f -Operation chat_prompt -Prompt "获取能力" -ClearHistory
```

脚本说明：

- `invoke-debug-runtime-operation.ps1` 对 `probe / capabilities / capture_and_read / swipe / exec_shell_limited / events` 统一走 `DebugRuntimeBridgeActivity`
- 当 `-Operation chat_prompt` 时，脚本会自动改走 `DebugRuntimeBridgeReceiver`
- `-Prompt` 通过 PowerShell 参数数组透传，能避免宿主机侧 `adb shell am` 把带空格 extras 拆参
- 结果会自动落到 `debug-artifacts/runtime/<operation>-result.json`，方便回归时留档

5. 如需手工确认，也可直接使用桥接入口执行 `probe`：

```bash
adb shell am start -W \
  -n com.clawdroid.app.debug/com.clawdroid.app.DebugRuntimeBridgeActivity \
  --es operation probe
adb shell run-as com.clawdroid.app.debug cat files/debug-runtime-result.json
```

6. 预期至少检查以下字段：
   - `finalState`
   - `stateTrace`
   - `degradedReason`
   - `packageName`
   - `socketName`
   - `signatureDigest`

7. 再依次执行：
   - `capabilities`
   - `capture_and_read`
   - `swipe`
   - `exec_shell_limited`
   - `events`

补充说明：

- 若直接使用 `adb shell am start/broadcast` 从宿主机传递带空格的 extras，`wm size`、`/shell wm size` 这类字符串会被宿主 shell 拆参
- 因此桥接 `exec_shell_limited` 复测时，优先使用 Activity 内置默认命令，或改用不含空格的提示词做聊天回归，例如 `获取能力`
- 若 CLI 复测结果只收到 `wm` 或 `/shell`，应先判定为 ADB 传参问题，而不是 Runtime 白名单或聊天逻辑失败

8. 对每项记录以下结果：
   - `ok`
   - 动作级返回值，如 `accepted / exitCode / timedOut / frameCount / closedReason`
   - 当前 `root / accessibility / lsposed / lsposedRuntimeLoaded`

9. 判断口径：
   - `finalState=Ready` 且动作链路通过：视为当前 App/模块组合已打通
   - `finalState=Degraded`：优先检查设备端实际生效的 `runtime.yaml` 能力开关与白名单
   - `finalState=Closed` 或桥接直接失败：优先检查当前 App 是否已获 Root、`auth.allowed_signatures` 是否与当前安装包匹配、模块端共享密钥是否已同步

10. 如需补做聊天业务闭环，可继续使用统一脚本，或显式广播 `DebugRuntimeBridgeReceiver`：

```bash
pwsh -File .\scripts\invoke-debug-runtime-operation.ps1 -DeviceId 7710a01f -Operation chat_prompt -Prompt "获取能力" -ClearHistory
```

手工命令如下：

```bash
adb shell am broadcast -n com.clawdroid.app.debug/com.clawdroid.app.DebugRuntimeBridgeReceiver \
  --es operation chat_prompt \
  --es prompt 获取能力 \
  --ez clear_history true
adb shell run-as com.clawdroid.app.debug cat files/debug-runtime-result.json
```

11. 聊天链路的通过口径：
   - `ok=true`
   - `chatBusy=false`
   - `assistantState="Final"`
   - `assistantMessage` 中包含工具执行结果摘要，例如 `Capabilities 成功` 或其他已知动作结果

### 前台截图快捷脚本

针对当前这台 MIUI 设备，项目已提供前台截图快捷入口，避免每次手写长命令：

```bash
pwsh -File .\scripts\capture-clawdroid-foreground.ps1 -DeviceId 7710a01f
```

常用别名：

```bash
pwsh -File .\scripts\capture-clawdroid-foreground-root.ps1 -DeviceId 7710a01f
pwsh -File .\scripts\capture-clawdroid-foreground-fast.ps1 -DeviceId 7710a01f
```

说明：

- `capture-clawdroid-foreground.ps1` 会固定 `com.clawdroid.app.debug` 与 `MainActivity`，并默认采用当前最稳的“自动拉起前台 + 跳过 UI dump 校验 + root 截图”策略
- `capture-clawdroid-foreground-root.ps1` 是最稳妥入口，适合保前台一致性
- `capture-clawdroid-foreground-fast.ps1` 默认走 `exec-out screencap`，适合更快的截图回归
- 若当前设备仍会被 MIUI 桌面抢前台，优先继续使用上述脚本，而不是重新拼接原始 `adb shell screencap` 命令

### 关键验证点

以下四项至少命中三项，且第 1 项必须成功，才可视为“真实注入通过”：

1. Marker 文件存在：

```bash
adb shell su -c 'cat /data/user/0/com.clawdroid.app.debug/files/xposed_runtime_marker.json'
```

2. Marker 内容中包含：

- `xposed_injected=true`
- `process_name` 为当前 App 进程
- `package_name` 与当前安装包名一致

3. App 侧本地环境检测显示：

- `LSPosed: injected`
- `LSPosed 进程` 非空

4. `Capabilities` 返回：

- `lsposed_runtime_loaded=true`
- `lsposed_runtime_process="<当前进程名>"`

5. 事件流中的 `capability_changed` 返回：

- `lsposed_runtime_loaded=true`

### 常见失败分层

若结果未通过，按以下层次排查：

- `lsposed=false`：
  说明设备未安装 LSPosed，或安装目录不完整
- `lsposed=true` 但 `lsposed_runtime_loaded=false`：
  说明框架存在，但模块未启用、未配置作用域，或未重新拉起目标进程
- `lsposed=true` 且设备上看不到标准 `LSPosed Manager` 包：
  先检查是否存在可替代的管理入口；若入口中看不到 `com.clawdroid.app.debug` 自身，通常不是 ClawApp 缺少 `xposed_init`，而是当前管理入口无法正确显示或编辑自作用域
- Marker 不存在：
  优先检查 `xposed_init`、入口类、包名匹配和 LSPosed 作用域
- Marker 存在但 `Capabilities` 仍为 `false`：
  优先检查运行时读取路径、包名是否一致，以及是否读取了错误用户目录

## 7. 失败判定

出现以下任一情况，判定本轮验收失败：

- `verify.sh` 返回非 `0`
- 新模块 `module.prop` 不是 `id=clawruntime`
- 新模块缺少 `clawdroid-runtime`
- 新模块缺少 `config/runtime.yaml`
- 模块安装后 `service.sh` 未启动运行时且日志无记录
- 新模块未生成 `webroot/status.json` 或 `webroot/verify.json`
- 发布包场景下 `auth.allowed_signatures` 仍为空
- 仓库模板文件中仍出现真实 `auth.shared_secret`
