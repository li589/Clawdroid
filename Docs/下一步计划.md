## 2. 原则

下一阶段 **先验收、再加深、后扩张**：

1. 不继续堆新域工具，直到 Assist MCP + 工具冒烟清单勾完一轮
2. 兼容矩阵至少再补一台非 MIUI 或非 Magisk 设备，再谈「稳定性里程碑」
3. Agent 深化优先打通 `task_*` 与聊天任务台，而不是再加供应商
4. Xposed 保持「页面身份 / 浅层 view」，不做微信深度自动化

## 3. 建议执行顺序

### P0 — 收口已落地能力（1–2 周）

目标：把 `9e4bba9` 扩张面变成可复现证据，而不是「代码存在」。

1. **Assist MCP 联调**  
   按 [assist-mcp.md](assist-mcp.md)「联调验收清单」逐项勾选；失败项记错误码与隧道状态。
2. **工具冒烟（同设备）**  
   覆盖 `file_*` / `app_*` / `download_*` / `notification_list` / `web_*` / `sandbox_shell` / `camera_*` / `shizuku_*`（有授权时）。
3. **Xposed 回归**  
   Settings + Browser 焦点/view 推送；确认 Overview live store；微信仅验证「默认关闭」。
4. **回填文档**  
   将本轮结果摘要写入兼容矩阵「回归批次」或本文件「进度日志」。

完成标准：Assist 清单 ≥80% 勾选；无 P0 级崩溃；已知限制有可检索描述。

### P1 — 稳定性与兼容（对应 release `v0.2`）

1. 第二台真机加入 [compatibility-matrix.md](compatibility-matrix.md)（优先非 MIUI 或 KernelSU/APatch）
2. CI：`app-compile` 增补 `:app:testDebugUnitTest`（密钥用现有 CI secret 占位）
3. 补齐 Overview / CapabilityProbe / ToolDispatcher 纯逻辑测试缺口
4. 关闭或归档 `debug-device-ui-review.md` 中已修复项，只保留未关闭假设

完成标准：矩阵 ≥2 行真实设备；CI 同时跑 Go test + App unit test；主路径无已知启动崩溃。

### P2 — 任务化 Agent 闭环（对应 release `v0.4` 核心）

1. 聊天页升级为任务执行台：状态、步骤、失败原因、取消
2. 打通 App Agent ↔ Runtime `task_submit/get/list/cancel`
3. 将 `AiAgentOrchestrator` 从固定短环升级为可取消、可等待信号的任务环
4. Overview 诊断卡继续语义化（签名不匹配 / Root 未放行 / caps stale）

完成标准：一次「用户自然语言 → 多步工具 → 可取消任务」真机演示可复现。

### P3 — Runtime 深化与适配治理（`v0.3` / `v0.5` 交错）

1. Runtime：文件桥写路径、shell 白名单、事件背压与限流 hardening
2. Xposed：适配器版本门与熔断审计；更多只读页面身份适配（仍禁止深度自动化默开）
3. 蓝图工具（`catalog.overlay.json` → `planned`）按需落地，避免无验收扩张

## 4. 明确暂缓

- PC 侧协助守护进程、scrcpy、摄像头实时预览 UI
- SFTP 密钥文件认证、GPU/NPU 算力调用

- 微信深度操控与任意第三方业务自动化
