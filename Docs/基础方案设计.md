# Clawdroid 项目架构设计说明书 (V4.0 最终定稿版)

## 一、项目定位

Clawdroid 是一套面向 Android 8 至 Android 16 的智能执行底座，目标是在 `Root + LSPosed` 环境下提供稳定的高权限执行能力，并在能力受限时退化为可接受的基础自动化能力。系统的核心价值不是单纯“拿到更高权限”，而是将模型决策、环境感知、特权执行与审计追踪拆分为可治理、可扩展、可降级的工程体系。

本方案的最终产品形态固定为两个交付物：

1. **Clawdroid App**

   - 一个标准 APK。
   - 承担 UI、任务编排、模型交互、无障碍能力、LSPosed 模块代码和 IPC 客户端。
2. **Clawdroid Magisk 模块**

   - 一个标准 Magisk 模块包。
   - 承担 Root 守护进程、启动脚本、配置、审计与高权限执行能力。

本方案的最终约束如下：

1. **只有两个部署单元**：不再拆分第三个独立程序。
2. **保留双组件主线**：继续采用 `ClawApp + ClawRuntime` 的双体架构，收敛高权限面。
3. **高生存性而非绝对保活**：工程目标是稳定与可恢复，而不是理论上的绝对常驻。
4. **最小暴露面**：Root 守护进程只提供受控本地 IPC，不直接暴露通用网络接口。
5. **谨慎使用系统注入**：默认优先无障碍和目标 App 精确适配，不默认依赖 `system_server` 深度注入。

## 二、总体设计原则

Clawdroid 后续版本遵循以下五条原则：

1. **最小特权原则**：任何高危操作都以能力项（Capability）形式显式授权，例如截屏、注入输入、文件读写、Shell 执行分别隔离。
2. **分层降级原则**：按照 `Root + Hook`、`Root Only`、`Accessibility Only`、`Observe Only` 四级能力自动降级，系统始终返回当前能力画像，不做隐式失败。
3. **可审计原则**：所有外部触发的执行请求必须生成结构化审计记录，能够追踪请求来源、参数摘要、执行结果、耗时与失败原因。
4. **协议优先原则**：`ClawApp` 与 `ClawRuntime` 的接口先定义协议与状态机，再实现具体进程和 UI，避免后期出现“功能能跑但协议失控”的耦合问题。
5. **应用内模块化原则**：与具体目标应用相关的感知和执行逻辑以 App 内部适配模块实现，不额外增加新的安装包或独立程序。

## 三、总体架构

Clawdroid 的架构分为“两个部署单元”和“三个运行时代码位置”。

### 1. 两个部署单元

| 维度     | Clawdroid App                                                | Clawdroid Magisk 模块                               |
| -------- | ------------------------------------------------------------ | --------------------------------------------------- |
| 分发形态 | 标准 APK                                                     | Magisk 模块 ZIP                                     |
| 技术栈   | Kotlin + Compose                                             | Go 静态编译 ELF + Shell                             |
| 主要内容 | UI、前台服务、无障碍、LSPosed 模块代码、任务编排、IPC 客户端 | ClawRuntime 守护进程、启动脚本、配置、审计、Root 执行能力 |
| 是否必须 | 是                                                           | 是                                                  |
| 升级节奏 | 相对高频                                                     | 相对低频                                            |

### 2. 三个运行时代码位置

1. **App 主进程**

   - 承载 UI、前台服务、无障碍、模型交互和 IPC 客户端。
2. **Root 守护进程**

   - 由 Magisk 模块拉起，承载高权限执行与审计。
3. **目标 App 内的 LSPosed 注入代码**

   - 来源仍然是同一个 APK，只是在目标应用进程中运行。
   - 该代码位置不构成第三个独立程序。

### 3. 代码组织分层

1. **App 交互层**

   - 负责用户界面、权限申请、任务编排、日志展示、连接状态展示。
2. **App 感知与适配层**

   - 由 `AccessibilityService`、可选 `MediaProjection`、可选 LSPosed 适配代码构成。
   - 负责 UI 树读取、窗口变化监听、通知读取、目标应用上下文提取与特定页面适配。
3. **Magisk 执行层**

   - 由 `ClawRuntime` 实现，对应源码目录为 `ClawRuntime/runtime`。
   - 仅暴露版本化本地 IPC 协议，不直接对外开启 TCP 监听。

## 四、进程模型与生命周期

### 1. ClawRuntime

- 由 Magisk 模块 `service.sh` 拉起，不表述为“init.d 级绝对保活”，而表述为“Root 守护进程，高于普通 App 生命周期约束”。
- 启动后完成以下动作：
  1. 自检运行环境。
  2. 建立本地 IPC 监听。
  3. 加载能力配置与允许的调用方信息。
  4. 输出健康状态与版本信息。
  5. 进入事件循环。

### 2. Claw-App

- 常驻前台服务只作为“交互稳定器”和“能力展示器”，不是保活真相来源。
- App 启动后先请求能力画像：
  - Root 是否可达。
  - ClawRuntime 是否在线。
  - Accessibility 是否已授权。
  - MediaProjection 是否已授权。
  - LSPosed 模块是否存在且已激活。
- UI 层只依据能力画像开放功能入口，避免用户点击后才发现不可用。

### 3. LSPosed 适配代码

- LSPosed 能力属于 `Clawdroid App` 的一部分，不单独作为新程序分发。
- 不默认注入系统所有关键进程。
- 采用白名单策略：只对明确需要的包名、版本区间、生效场景进行激活。
- 每个目标应用适配模块具备独立开关、版本约束和熔断标记。

## 五、权限与安全边界设计

### 1. Root 守护进程的最小暴露面

ClawRuntime 不再直接承担“本地 Web 控制台 + 通用 REST API + WebSocket 全开放入口”。推荐调整为：

- **唯一正式入口**：本地 UNIX Domain Socket。
- **协议类型**：建议使用长度前缀帧 + JSON 或 Protobuf。
- **动作模型**：仅允许固定枚举动作，例如：
  - `ping`
  - `get_capabilities`
  - `capture_screen`
  - `inject_tap`
  - `inject_swipe`
  - `read_file_limited`
  - `exec_shell_limited`
  - `subscribe_events`
- **拒绝设计**：默认拒绝未知动作、超范围参数、超长负载与危险 Shell。

如果需要本地调试面板，应由 `ClawApp` 代理展示，或由 Magisk 模块输出本地静态诊断快照，而不是让 `ClawRuntime` 自带长期开放的管理控制台。

### 2. 调用方鉴权

仅检查 `UID == Claw_UID` 不足以满足安全要求，建议同时做四层校验：

1. **对端凭证校验**：通过 `SO_PEERCRED` 或等效机制校验 PID / UID。
2. **包名映射校验**：由 App 侧上报包名与签名摘要，`ClawRuntime` 对照安装信息或预置白名单。
3. **会话握手校验**：首次握手使用一次性随机挑战，避免简单重放。
4. **能力令牌校验**：不同指令带不同能力标签，例如“只读观察令牌”和“输入注入令牌”分离。

### 3. 审计与回放防护

每条指令至少记录以下字段：

- `request_id`
- `session_id`
- `caller_uid`
- `action`
- `arg_digest`
- `started_at`
- `ended_at`
- `result_code`
- `error_message`

所有高危动作必须可回溯，便于后续做安全审查和异常定位。

## 六、SELinux 与 Root 兼容策略

### 1. 对原策略的修订意见

原方案中直接对多个 `untrusted_app_*` 域进行广泛 `allow` 的思路实现成本低，但存在三类问题：

1. **放权范围过大**：容易让同域其他 App 获得接近的连接能力。
2. **跨 ROM 不稳定**：不同厂商域划分与策略组合差异明显，维护成本高。
3. **难以审计**：后续难判断到底是 App 主进程、LSPosed 注入代码还是恶意同域进程发起的访问。

### 2. 推荐策略

按优先级采用以下兼容路径：

1. **首选：最小化 live sepolicy 修补**

   - 仅为目标 socket 通信放开必要权限。
   - 尽量不做宽泛的 `untrusted_app -> magisk` 全域授权。
2. **次选：将高危动作收敛为 Brain 主动执行**

   - Claw 仅发送抽象命令，不直接传递文件描述符或复杂对象。
   - 这样可以减少对 `fd use` 等策略的依赖。
3. **再次选：KernelSU / APatch 下使用更可控的特权边界**

   - 但仍应避免“全局放行”式 Hook。
   - 推荐做路径、进程、动作三元组粒度的控制。
4. **兜底：禁用特定高危能力**

   - 当策略注入不可行时，自动关闭输入注入或文件桥接，而不是让整个系统不可用。

### 3. 结论

SELinux 的目标不是“完全突破”，而是“以最小修改让必要能力可达”。在产品层面应允许能力缺失，而不是要求所有设备都达到同一特权水平。

## 七、本地 IPC 设计

### 1. 通信选型

保留 UNIX Domain Socket 是正确的，但建议明确为：

- 使用 **抽象命名空间 UDS** 或 **文件路径 UDS** 二选一。
- 若追求无需文件落地与简单部署，可用抽象命名空间。
- 若追求额外的文件权限与上下文控制，可用文件路径 UDS。

建议优先实现抽象命名空间方案，后续为特定环境保留文件路径方案。

### 2. 协议框架

请求帧建议如下：

```json
{
  "version": 1,
  "request_id": "uuid",
  "timestamp": 1710000000,
  "action": "inject_tap",
  "capability": "input.inject",
  "args": {
    "x": 540,
    "y": 1200,
    "display_id": 0
  }
}
```

响应帧建议如下：

```json
{
  "request_id": "uuid",
  "ok": true,
  "code": 0,
  "message": "success",
  "data": {
    "latency_ms": 12
  }
}
```

### 3. 关键协议要求

- 支持版本号，便于以后升级协议。
- 支持超时与取消。
- 支持服务端主动事件推送，如窗口变化、任务状态变化、守护进程重启。
- 同一 `request_id` 保证幂等，避免重试导致重复点击或重复执行 Shell。

## 八、执行能力设计

ClawRuntime 推荐拆成以下子模块：

1. **Input Executor**

   - 优先使用稳定、可兼容的输入注入方式。
   - 统一封装点击、长按、滑动、文本输入、按键事件。
2. **Capture Executor**

   - Root 截屏与 MediaProjection 截图并存。
   - 由能力协商决定优先级。
3. **Shell Executor**

   - 只提供受限命令模板，不直接开放任意 Shell。
   - 示例：`am start`、`pm list packages`、`dumpsys window windows`。
4. **File Bridge**

   - 仅开放白名单路径和只读能力，写能力需单独能力项批准。
5. **Task Scheduler**

   - 把单次动作和多步骤工作流统一建模。
   - 支持暂停、恢复、重试、取消。

## 九、感知能力设计

感知层建议分三级：

1. **基础感知**

   - Accessibility 获取窗口、控件、文本、焦点状态。
   - 覆盖绝大多数系统级 UI 自动化场景。
2. **增强感知**

   - MediaProjection 获取更高保真截图。
   - 配合 OCR、图像定位与模型理解。
3. **定制感知**

   - App 内置的 LSPosed 适配代码针对特定 App 提供更稳定的语义级信息，如会话列表、消息结构、页面状态。

不建议把“高阶感知”默认建立在对 `system_server` 的深度注入上，因为这会显著放大 ROM 差异、崩溃风险与维护成本。

## 十、模型编排与任务流

建议将“大模型决策”和“移动端执行”之间增加一个轻量任务规划层：

1. 模型输出高层意图，例如“打开微信并给某联系人发送消息”。
2. 规划器将意图拆成平台动作，例如“启动应用”“等待页面稳定”“定位联系人”“点击”“输入文本”“确认发送”。
3. 执行器逐步执行并持续回传状态。
4. 若中途环境与预期不一致，则回退到规划器重新决策。

这样可以避免模型直接生成底层设备指令，降低误操作与不可控行为。

## 十一、能力分级与降级矩阵

| 模式         | Root | Accessibility | LSPosed | 可用能力                           |
| ------------ | ---- | ------------- | ------- | ---------------------------------- |
| A. Full      | 是   | 是            | 是      | 感知 + 高精执行 + 目标应用深度适配 |
| B. Root Only | 是   | 否            | 否      | 截图、Shell、有限输入注入          |
| C. Assistive | 否   | 是            | 否      | 无障碍自动化、基础感知             |
| D. Observe   | 否   | 否            | 否      | 仅 UI、模型、日志、远程编排        |

产品界面应显式展示当前所处模式，避免用户对能力产生误判。

## 十三、测试与观测性设计

原草案缺少验证闭环，这会在高版本 Android 适配时成为主要风险。建议新增以下体系：

### 1. 测试

- **协议测试**：ClawRuntime 与 Claw 的序列化、错误码、超时、幂等测试。
- **设备集成测试**：至少覆盖 Android 8、10、12、14、16 各一台或等价镜像。
- **适配回归测试**：每个目标 App 适配模块维护版本适配清单。
- **故障注入测试**：模拟 ClawRuntime 重启、Socket 断连、权限被回收、无障碍被系统关闭等情况。

### 2. 观测性

- ClawRuntime 输出结构化日志，支持按 `request_id` 查询。
- App 端展示最近执行记录、能力画像、错误诊断建议。
- 增加健康检查接口：`ping`、`version`、`capabilities`、`last_error`。

## 十四、分阶段落地路线

### Phase 1：最小可用底座

- 完成 Brain 守护进程、UDS 协议、能力查询、点击/滑动/截图。
- App 完成连接、状态展示、手动动作调试。
- 使用 Accessibility 作为默认感知层。

### Phase 2：任务化执行

- 引入任务调度器、幂等请求、取消与重试。
- 加入结构化审计日志与错误码体系。
- 建立设备兼容矩阵。

### Phase 3：目标应用深度适配

- 接入 LSPosed 适配能力。
- 为微信、浏览器等目标 App 提供 App 内部适配模块。
- 按适配模块做熔断与版本治理。

### Phase 4：模型闭环

- 引入任务规划器。
- 加入 OCR / 截图理解 / 页面语义反馈。
- 支持云端模型与端侧小模型切换。

## 十五、关键风险清单

1. **ROM 差异风险**：同版本 Android 在不同厂商 ROM 下行为可能完全不同。
2. **Hook 稳定性风险**：目标 App 升级后 App 内置适配代码极易失效。
3. **SELinux 风险**：不同提权方案对策略注入能力差异很大。
4. **误操作风险**：模型直接驱动输入注入可能造成不可逆点击。
5. **攻击面风险**：Root 守护进程一旦暴露过多 API，将成为整机高危入口。

因此，Clawdroid 的设计目标应从“追求全能”转向“能力分级、最小暴露、渐进增强、可观测可熔断”。

## 十六、实施前置项

在进入编码阶段前，项目应先完成以下前置项：

1. **协议冻结**

   - 冻结 `protocol.md` 中的请求响应结构、错误码和状态机。
2. **威胁模型确认**

   - 明确调用方身份、信任边界、Root 侧攻击面、LSPosed 注入边界与日志合规要求。
3. **兼容矩阵建立**

   - 至少覆盖 Android 大版本、Root 方案、典型 ROM 和目标 App 主版本。
4. **MVP 范围锁定**

   - 第一阶段仅实现连接、能力查询、截图、点击、滑动、基础无障碍感知和日志查看。

该前置项完成后，方案即可进入实现阶段。

## 十七、细节设计补充

### 1. 配置分层设计

为避免配置散落在 App、适配代码与守护进程中，建议采用四层配置模型：

1. **编译期默认配置**

   - 固化在 `ClawApp` 与 `ClawRuntime` 的默认常量中。
   - 仅保存协议版本、默认超时、默认日志级别等稳定参数。
2. **设备级运行配置**

   - 存放于 `ClawRuntime` 可访问的本地配置文件。
   - 用于定义 Socket 名称、审计目录、特性开关、路径白名单、动作限流值。
3. **应用级用户配置**

   - 由 Claw-App 保存。
   - 用于保存模型供应商、界面偏好、目标应用适配开关、任务模板。
4. **会话级临时配置**

   - 只存在于一次任务或一次连接周期内。
   - 用于存储当前能力画像、会话令牌、临时超时和本次任务上下文。

建议配置优先级为：`会话级 > 应用级 > 设备级 > 编译期默认值`。

### 2. ClawRuntime 配置建议项

ClawRuntime 侧建议统一维护一份结构化配置，最少包含以下字段。当前正式配置以 `runtime`、`auth`、`capability`、`paths` 四个分区为准：

```yaml
runtime:
  socket_name: "clawdroid_secure_ipc"
  protocol_version: 1
  request_timeout_ms: 8000
  max_payload_bytes: 262144
  rate_limit_per_minute: 120
  log_level: "info"
  audit_dir: "/data/local/tmp/clawdroid/audit"
auth:
  shared_secret: "REPLACE_WITH_RELEASE_SECRET"
  allowed_packages: "com.clawdroid.app,com.clawdroid.app.debug"
  allowed_signatures: "SHA256:TODO"
  timestamp_skew_seconds: 120
  challenge_ttl_seconds: 30
  session_ttl_seconds: 300
capability:
  input_inject_enabled: true
  screenshot_enabled: true
  shell_enabled: true
  file_bridge_enabled: true
paths:
  readonly_whitelist: "/sdcard/Pictures,/sdcard/Download"
```

### 3. 连接状态机

ClawRuntime 与 Claw 的连接流程建议显式定义状态机，避免后续出现“连接已建立但未完成认证”的灰色状态。

推荐状态如下：

1. `Disconnected`
2. `SocketConnected`
3. `PeerVerified`
4. `ChallengeIssued`
5. `Authenticated`
6. `CapabilitySynced`
7. `Ready`
8. `Degraded`
9. `Closed`

状态流转建议：

- 连接建立后先校验对端凭证，再进入挑战应答。
- 认证通过后立即同步能力画像与协议版本。
- 任一关键能力缺失时不强制断开，可进入 `Degraded` 状态继续提供低能力服务。
- 会话超时、令牌失效、协议版本不兼容时应立即进入 `Closed`。

### 4. 任务状态机

任务执行同样建议统一状态机，供 App、ClawRuntime、适配代码、日志查询共用：

1. `Created`
2. `Queued`
3. `Running`
4. `WaitingSignal`
5. `Retrying`
6. `Succeeded`
7. `Failed`
8. `Cancelled`
9. `Compensating`

其中：

- `WaitingSignal` 用于等待页面稳定、权限授权、窗口切换、外部事件回调。
- `Retrying` 用于可恢复失败，例如目标节点未出现、页面还未完成渲染。
- `Compensating` 用于执行补偿动作，例如返回桌面、关闭弹窗、恢复初始页面。

### 5. 错误码设计

建议错误码分段，避免后续每个模块各自定义一套不可维护的编号。

| 错误码段 | 范围        | 含义                                       |
| -------- | ----------- | ------------------------------------------ |
| 通用错误 | 0 - 999     | 协议、参数、超时、取消等基础错误           |
| 认证错误 | 1000 - 1999 | 包名、签名、令牌、会话、挑战失败           |
| 能力错误 | 2000 - 2999 | Root 不可用、无障碍未授权、截图能力缺失    |
| 执行错误 | 3000 - 3999 | 输入注入失败、Shell 执行失败、文件桥接失败 |
| 适配错误 | 4000 - 4999 | Hook 失效、版本不匹配、目标页面结构变化    |
| 系统错误 | 5000 - 5999 | 守护进程异常、ROM 不兼容、SELinux 拒绝     |

推荐预置以下基础错误：

| 错误码 | 名称                              | 含义               |
| ------ | --------------------------------- | ------------------ |
| 0      | `OK`                            | 成功               |
| 1      | `ERR_UNKNOWN`                   | 未分类错误         |
| 2      | `ERR_INVALID_REQUEST`           | 请求结构非法       |
| 3      | `ERR_UNSUPPORTED_VERSION`       | 协议版本不兼容     |
| 4      | `ERR_TIMEOUT`                   | 请求超时           |
| 5      | `ERR_CANCELLED`                 | 请求被取消         |
| 1001   | `ERR_PEER_VERIFY_FAILED`        | 对端凭证校验失败   |
| 1002   | `ERR_SIGNATURE_MISMATCH`        | 签名摘要不匹配     |
| 2001   | `ERR_ROOT_UNAVAILABLE`          | Root 不可用        |
| 2002   | `ERR_ACCESSIBILITY_UNAVAILABLE` | 无障碍未授权       |
| 3001   | `ERR_INPUT_INJECT_FAILED`       | 输入注入失败       |
| 3002   | `ERR_SHELL_DENIED`              | Shell 动作未被允许 |
| 5001   | `ERR_SELINUX_DENIED`            | SELinux 拒绝       |

### 6. 协议动作约束

动作接口建议采用“固定动作 + 固定参数模式”，每个动作都要声明以下约束：

- 参数结构。
- 参数范围。
- 是否幂等。
- 是否可取消。
- 默认超时。
- 审计级别。
- 失败后的补偿策略。

例如 `inject_tap` 的约束建议如下：

- 参数：`x`、`y`、`display_id`
- 范围：坐标不得超出当前显示区域
- 幂等：否
- 可取消：否
- 默认超时：1000 ms
- 审计级别：高
- 补偿策略：无，失败时只记录错误并中止当前步骤

### 7. 目标应用适配元数据

每个目标应用适配模块建议维护最小清单文件，用于在运行时判断是否可加载：

```yaml
adapter_id: "wechat"
adapter_version: "1.0.0"
target_packages:
  - "com.tencent.mm"
target_app_versions:
  min: "8.0.0"
  max: "8.0.99"
required_capabilities:
  - "observe.ui"
  - "input.inject"
hooks:
  - "LauncherUI"
  - "ChattingUI"
degrade_policy: "disable_adapter"
```

运行时应检查：

- 目标包是否存在。
- 目标版本是否在兼容范围内。
- 所需能力是否具备。
- 上一次是否被熔断。

### 8. 审计日志格式

建议采用按行 JSON 的审计文件格式，便于后续检索与上传：

```json
{
  "ts": "2026-06-23T10:00:00Z",
  "request_id": "2e4f...",
  "session_id": "a19b...",
  "caller": "com.clawdroid.app",
  "action": "inject_tap",
  "args_digest": "sha256:...",
  "capability": "input.inject",
  "result_code": 0,
  "latency_ms": 12
}
```

日志建议区分三类：

1. **审计日志**：记录谁调用了什么动作。
2. **运行日志**：记录守护进程状态与错误堆栈。
3. **任务日志**：记录工作流步骤推进情况。

### 9. 目录与数据落盘建议

为降低后续迁移成本，建议预留统一目录：

```text
/data/local/tmp/clawdroid/
  run/
    runtime.sock.meta
  audit/
    2026-06-23.audit.jsonl
  logs/
    runtime.log
  cache/
    screenshots/
    task/
  config/
    runtime.yaml
```

如果运行环境不适合使用上述目录，应通过设备级配置覆盖，而不是在代码里写死路径。

### 10. 限流与熔断建议

为防止模型失控、适配代码异常或错误重试导致频繁注入，建议引入以下保护：

- 输入注入动作按会话限流，例如每分钟不超过 60 次。
- Shell 动作默认关闭批量执行。
- 同一适配模块连续失败达到阈值后进入熔断状态。
- 同一页面识别失败超过阈值后自动降级为人工确认。

### 11. 版本协商策略

ClawRuntime 与 Claw 建议采用“主版本必须一致，次版本向后兼容”的协商方式：

- 主版本不一致时拒绝执行高危动作。
- 次版本不一致时允许连接，但只开放双方都认识的动作集。
- 协议协商结果应写入当前会话状态，供日志与调试界面显示。

## 十八、历史记录

### V1.0 基础草案

- 定义了早期 `Clawdroid-Brain` 与 `Clawdroid-Claw` 双组件主线。
- 提出了 Root 守护进程、LSPosed 注入、无障碍降级、SELinux 打通、本地 IPC 等核心方向。
- 存在的问题是：原则强、实现细节少，且对系统注入范围、保活能力、SELinux 放权范围表述偏激进。

### V2.0 施行版

- 进一步强化了 Root 守护、特权隔离、抽象命名空间 UDS、动态 SePolicy 注入等思路。
- 明确了运行时侧使用 Go、App/Claw 侧使用 Kotlin + Compose 的技术栈。
- 主要不足是：仍偏重“打通能力”，尚未充分收敛攻击面，也缺少协议治理、状态机、错误码、测试矩阵和目标应用适配治理设计。

### V3.0 完善版

- 在当时版本中，将整体架构重构为 `Agent/UI`、`感知层`、`执行层`、`插件层` 四层模型。
- 将“绝对保活”修订为“高生存性”，将“广泛放权”修订为“最小特权 + 自动降级”。
- 强化了 IPC 鉴权、能力分级、工程化分层、测试与观测性、分阶段落地路线。

### V3.1 细化版

- 将修正历史统一整理到文档末尾，方便后续持续追加。
- 新增了配置分层、连接状态机、任务状态机、错误码分段、适配元数据、目录落盘、限流熔断和版本协商等可编码细节。
- 文档从“架构方向说明”进一步推进到“可指导实现的设计说明”。

### V3.2 收敛版

- 明确部署形态固定为 `1 个 App + 1 个 Magisk 模块`，不再让分层描述造成“三个程序”的误解。
- 将“插件层”统一改写为 App 内部的目标应用适配模块，LSPosed 代码被定义为同一 APK 的组成部分。
- 区分“部署单元”和“运行时代码位置”，保留工程分层但收敛外部交付复杂度。

### V4.0 最终定稿版

- 将正文整体改写为“最终方案口径”，不再在主文中保留评估修订式表述。
- 合并并压缩“总体架构”和“部署边界”中的重复解释，统一为两个部署单元、三个运行时代码位置。
- 将“最终建议”重构为“实施前置项”，使文档更适合直接进入研发执行阶段。
